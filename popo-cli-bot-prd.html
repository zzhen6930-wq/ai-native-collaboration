PRD · POPO 机器人能力

通过 CLI 使用 POPO 机器人发消息

通过新增 Skill 让 Agent 学会"用机器人身份发消息"，本期只做发消息，不做创建、不做回调，让用户最快用起来。

文档版本：V2.0 作者：青远(张振) 更新日期：2026-06-11 状态：待评审

1背景与价值

当前问题

POPO CLI 当前只能用用户本人的身份发消息。这带来三个核心问题：

  • 用户呼声 ①（执行遇阻通知）：当 Agent 执行遇阻时，能通过机器人发消息提醒自己，然后人工介入处理问题。
  • 用户呼声 ②（任务完成播报）：当前有很多服务类机器人，当 Agent 完成各种任务后，业务希望通过不同的机器人发消息提醒给个人或群里。
  • 群内身份诉求：很多用户向群里发消息时希望使用机器人身份，而不是自己的身份，以免被同事误以为是用户本人发言。

真实用户反馈示例

解决方案

核心思路（V2 调整）

本期不在本地绑定机器人——为了发消息这一个动作让用户去扫码创建 / 绑定机器人对用户来说太重。改为：在安装 CLI 时附带一个新 Skill，由它教会 Agent "如何用机器人身份发消息"——用户在指令里直接说出机器人名称，Agent 自动完成"搜索机器人 → 校验权限 → 调用开放平台发消息"。

用户价值

  • ✅ 用户零配置：不需要扫码、不需要本地绑定，安装 CLI 即可使用。
  • ✅ 群内身份合规：用机器人身份发消息，避免被同事误以为是用户本人发言。
  • ✅ 满足用户呼声：补齐 CLI 用户群反馈最多的"机器人发消息"能力。

2典型场景

场景 1 · 群内身份合规
用机器人身份在群里发通知

用户对 Agent 说："让 [运维助手] 在 XX 群里发条消息：今晚 22:00 维护，预计 30 分钟。" Agent 用运维助手机器人身份发出，群成员一眼看出"这是机器人发的官方通知"，不会误以为是用户本人发言。

场景 2 · 任务结果播报
Agent 任务完成后机器人推送结果

长任务（部署、跑数据、生成报告）完成后，Agent 通过用户指定的机器人推送一条"任务已完成 + 结果链接"给指定的人或群。

场景 3 · 多 Agent 各司其职
不同 Agent 用不同机器人

用户已有多个机器人（如"代码助手"、"日报助手"、"运维助手"），不同 Agent 在不同任务里使用不同机器人发消息，消息来源一目了然，不会串台。

场景 4 · 转发 / 通知放大
机器人代为推送

某些通知不适合用个人身份发（避免对方误认为是私人请求），但又希望对方收到。借机器人身份发出，更显正式。

3能力总览

本期交付一个能力：在安装 CLI 时携带一个新的 Skill，让 Agent 知道"如何用机器人身份发消息"。完整运行链路如下：

阶段动作
① 用户指令 用户对 Agent 说："让 [机器人名称] 给 XX 发消息：YYY"，必须明确机器人名称。
② 搜索机器人 Skill 引导 Agent 按"机器人名称"搜索，找到目标机器人。
③ 权限校验 双重判定：a) 当前用户是该机器人的开发者或管理员；b) 接收人在该机器人可使用范围内。
④ 调接口发送 校验通过 → 用机器人身份调用 POPO 开放平台发消息接口，消息以机器人身份送达接收方。

4Skill 设计

定位

  • 新增一个"机器人发消息" Skill，随 CLI 安装一并下发，用户无需手动启用。
  • Skill 的职责是告诉 Agent 怎么做：调用顺序、必要参数、权限判断点、错误处理。
  • 本 Skill 与已有的"IM Skill 给机器人发消息"是互补关系（详见 §7）。

用户使用方式

用户
让 "代码助手" 在「研发周报群」里发条消息：本周代码合并已完成，详细见 xxx。
AGENT
已通过机器人「代码助手」在"研发周报群"发送消息 ✅
机器人
（在群里）代码助手：本周代码合并已完成，详细见 xxx

关键约束

项说明
必须指定机器人名称 用户指令必须明确说明用哪个机器人发送（如"用 [运维助手] 发"）。Agent 不主动猜测、不默认选机器人。
消息内容能力 支持文本、图片、文件三类内容，与当前 CLI 已支持的发消息能力保持一致。
发送方显示 消息送达时，发送方显示为机器人名称 + 头像，不显示用户本人身份。
一次成行 Skill 内部把"搜机器人 → 校权限 → 发消息"封装为顺序动作，避免 Agent 来回试错、多次工具调用。

权限校验（两道闸门）

  • ① 操作机器人的权限：当前 CLI 用户必须是该机器人的开发者或管理员，否则 Skill 拒绝执行并向用户明确告知"您不是机器人 X 的开发者或管理员，无权操作"。
  • ② 接收方在可使用范围内：接收消息的用户 / 群必须在该机器人的"可使用范围"内，否则拒绝发送并告知"接收方不在机器人 X 的可使用范围"。
  • 两个权限均通过POPO 开放平台已有机制判定，不在本地维护权限信息。

5端到端用户流程

1
用户向 Agent 下达指令
用户在 Agent 对话中说出指令，明确指定机器人名称、接收方和消息内容。例："让 [代码助手] 给张三发消息：今晚部署 22:00 开始"。
2
Skill 引导 Agent 搜索机器人
Skill 让 Agent 按"机器人名称"搜索 POPO 机器人，定位到目标机器人。
3
权限双重校验
校验：① 当前用户能操作该机器人发消息；② 接收方在机器人可使用范围内。任一不满足，立即终止并告知用户。
4
调用开放平台发送
校验通过 → 以机器人身份调用 POPO 开放平台发消息接口，消息以机器人名义送达接收方。
5
向用户反馈结果
Agent 告知用户"已通过机器人 X 发送给 Y"，或在失败时给出明确原因（无权限 / 不在可使用范围 / 接口异常等）。

6本期不做

扫码创建 / 绑定机器人
为发消息这一个动作让用户去扫码创建机器人太重，本期不做。用户使用前提：自己已在 POPO 客户端拥有可用机器人。
本地保存机器人配置
不在本地存机器人 ID / app_id 等任何配置信息，每次按名称即时搜索，权限实时校验。
消息回调 / 双向对话
本期只做"Agent → 机器人 → 接收方"单向发送。用户回复机器人的消息不回推 Agent，后续视用户呼声评估是否补做。
默认机器人 / 自动选择
用户必须在指令中显式指定机器人名称，Skill 不做"猜默认机器人"逻辑，保持行为可预期。

7与已有 IM Skill 的关系

本 PRD 与之前「IM Skill 支持给机器人发消息」的需求是互补关系，方向相反，合在一起构成"AI ↔ 机器人 ↔ 人"的完整发送闭环：

维度IM Skill 给机器人发消息本 PRD（机器人发消息）
谁是消息发送方 用户本人 机器人（Agent 借机器人身份）
谁是消息接收方 机器人 普通用户 / 群
典型用法 "让 KM 小助手帮我搜一下 X" "让 [代码助手] 在群里通知 X"

8验收标准

#验收项判定标准
V1 Skill 随 CLI 安装下发 用户安装 CLI 后，无需任何手动操作即可在 Agent 中使用"用机器人发消息"能力。
V2 按名称定位机器人 用户指令包含机器人名称时，Skill 能引导 Agent 准确找到该机器人；同名机器人或找不到时给出明确提示。
V3 权限双重校验 用户无操作权限 / 接收方不在可使用范围时，立即终止并向用户明确说明原因，不发送消息。
V4 消息成功发送 校验通过后，消息以机器人身份（机器人名称 + 头像）送达接收方；支持文本 / 图片 / 文件三种内容。
V5 一次调用完成 Skill 内部把"搜索 + 校验 + 发送"作为一次顺序动作完成，Agent 不需要在中间多次试错。
V6 用户零配置 用户除"在 POPO 客户端已有可用机器人"外，无需任何额外配置（无扫码、无本地绑定、无 webhook）。
