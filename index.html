1背景与定位

1.1 背景

随着 AI Agent 在产品部门的应用扩展，多个垂类场景已浮现：

场景典型 Agent
用户问题反馈群运维群运维 Agent
需求评审评审 Agent
测试 / 发布反馈体验官 Agent
开发对接开发 Agent
未来扩展PM Agent / 运营 Agent / ...

1.2 核心问题

如果每个 Agent 都从 0 搭建，会导致 群消息监听、知识库读写、人工审核、定时调度等能力被重复建设。

1.3 方案核心思路

平台共性 Skill + 每个 Agent 一份 Prompt + 多维表配置参数
  • 平台共性 Skill：把跨场景重复的能力抽象为 9 个标准 Skill，由开发统一实现
  • 每个 Agent 一份 Prompt：业务规则、流程编排、判断逻辑、输出风格写在 Prompt 里
  • 多维表配置参数：分类映射、模板内容、关键字词典等结构化数据放在多维表，可灵活编辑

新场景接入策略：

  • 大部分新场景能通过 复用现有 Skill + 垂类场景 Prompt + 调整多维表配置 快速接入
  • 遇到全新能力（如调用某外部系统）时，仍需新增对应 Skill，但保持 Skill 与场景解耦

1.4 三者职责切分

为避免后续维护时出现"同一件事多处定义"，三层职责必须清晰：

📜
系统 Prompt
业务规则、流程顺序、判断逻辑、输出风格
👤 产品 直接写
📊
多维表配置
结构化数据：分类→人映射、模板内容、关键字词典
👤 产品 直接改表
🧩
9 个原子 Skill
标准化能力实现：建群、查知识、写表、推送...
👨‍💻 开发同学

判断归属的简单规则：

  • 文字描述的"规则、流程"？→ 系统 Prompt
  • 表格化的"规则、映射、清单"？→ 多维表
  • "如何执行"？→ 原子 Skill

3平台共性能力（9 大 Skill）

A. 群协作 A 类 · 3 项

A1 · 群协作
群运维
建群 / 拉人 / 改群名 / 解散群 / 群成员管理。任何"一事一群"的场景都用它。
典型复用评审一需求一群、体验官一招募一群、未来策划评审群
A2 · 群协作
私聊确认
给指定用户私聊推送内容 → 等待回复 → 处理决策（确认 / 拒绝 / 修改）。
典型复用评审定时私聊产品、体验官私聊确认进群、群运维私聊更新知识
A3 · 群协作
内部群审核闭环
AI 产出内容 → 推送到内部群 → 多人逐一确认 → 通过 / 拒绝 / 修改 → 落地。
典型复用群运维知识沉淀、评审意见汇总、反馈分类确认

B. 知识 / 数据 B 类 · 3 项

B1 · 知识/数据
知识库读写
跨知识源（Echo / KM / 代码库 / POPO 文档）的统一读写接口。
典型复用所有"知识检索 + 改写回复"或"知识沉淀"的场景
B2 · 知识/数据
多维表读写
标准化的多维表 CRUD + 状态流转能力。同时承担"读取产品配置的多维表参数"。
典型复用群运维工单表、评审意见表、体验官反馈表
B3 · 知识/数据
内容识别抽取
从一段文本中识别出"有价值的结构化内容"。
典型复用识别提问意图、识别评审意见、识别 Bug/卡点/需求
⚠️ 关键：B1 解决"租户下知识库不支持隔离"的痛点，本 Skill 提供逻辑隔离（按命名空间区分），消除"读跨场景知识"的风险。

C. 调度与触达 C 类 · 3 项

C1 · 调度
定时调度
跨场景的定时任务能力（每日 / 每周 / 自定义 cron）。
典型复用群运维日报、评审定时确认、体验官每晚整理
C2 · 调度
Agent 协同
Agent 之间的消息流转 / 任务移交。
典型复用评审 → 开发交付 Spec、体验 → 评审反馈新需求
C3 · 调度
模板化输出
标准化的输出格式（卡片 / 日报 / 候选知识 / 等）模板。
典型复用所有标准化输出（替代原"多维表字段、回复格式、智能分类映射"等专属配置）
💡 关键设计："多维表字段、回复格式、智能分类映射、能力依赖表"等内容均被抽象为 C3 模板化能力的实例。每个垂类 Agent 通过 Prompt 引用模板 + 在多维表填写模板内容即可，无需重复定义专属字段。

4多 Agent 协同关系

4 个 Agent 不是孤立的，而是通过 C2 Agent 协同 Skill 互通消息：

📋 产品需求触发
↓
🤖 评审 Agent
  • 群内记录评审意见
  • 私聊确认
  • 生成 Spec
→ Spec
💻 开发 Agent
  • （后续接入）
↑ 反馈新需求
📣 体验官 Agent
  • 招募 / 拉群
  • 识别有价值反馈
  • 每晚整理
↑ 反馈表互通
🛠️ 群运维 Agent
  • 答疑 / 建单
  • 知识沉淀
  • 日报

核心机制说明：

  • 4 个 Agent 不是孤立的，通过 C2 Agent 协同 Skill 互通消息
  • 每个 Agent 只关注自己的业务流程，跨 Agent 协同走 Skill 调用
  • 例如：体验官识别到"用户报了 Bug" → 通过 C2 把消息推给评审 Agent → 评审 Agent 自动评估是否要进需求池

Skill用途
A3 内部群审核闭环候选知识送审
B1 知识库读写检索 + 沉淀知识
B2 多维表读写工单跟进表 + 读取配置参数
B3 内容识别抽取识别提问意图
C1 定时调度每日 18:00 日报
C3 模板化输出回复 / 工单 / 日报 模板
▌ 系统 Prompt 内容（描述"做什么、怎么判断"） 
你是 POPO CLI 群运维助手，负责在用户反馈群里答疑、建单、沉淀知识。

【业务规则】
- 知识库范围：仅检索 popo-cli 命名空间，不读跨场景知识
- 命中阈值：检索置信度 ≥ 0.75 才自动答复，否则建单
- 输出风格：回复简洁专业

【流程编排】
① 收到 @机器人 消息 → 用 B3 识别意图（提问 / 闲聊 / 反馈）
② 提问意图 → 用 B1 在 popo-cli 知识库检索
③ 命中且阈值达标 → 用 C3 渲染回复模板 → 群内回复用户
④ 未命中 → 用 B2 创建工单 + 读多维表分类映射 → 自动指派处理人
⑤ 工单状态变为「已解决」→ 用 A3 推送候选知识到内部运维群
⑥ 运维确认后 → B1 写入知识库 + B2 标记「已沉淀」
⑦ 每天 18:00 由 C1 触发 → C3 渲染日报 → 推送内部群

【人工兜底】
- 用户回复「转人工」时，直接走步骤④
- 同一用户连续 2 次表达不满意时，自动走步骤④

你是需求评审助手，负责一需求一群的评审过程管理。

⑦ Spec 完成 → C2 协同移交给开发 Agent

📌 核心观察：两个 Agent 引用的 Skill 90% 重叠，差异只在"系统 Prompt 描述的业务规则"和"多维表里填的参数"。这就是标准化的价值：新场景接入 = PM 写一份 Prompt + 填一套多维表参数。

6群运维场景：首个落地 Demo

群运维 Agent 作为平台首个验证场景，验证 Skill 矩阵的可用性和复用性。

6.1 触发方式

阶段模式说明
V1@机器人 触发可控、不打扰，验证准召率
V2群内主动监听V1 准召率达标后切换

6.2 用户体验流程

用户在群里 @机器人："popo-cli 安装失败怎么办？"
Agent 立即基于知识库答复
用户继续追问"还是不行"，或点击"建单跟进"
Agent 判断答不准 → 触发建单 → 用户补充完整上下文 → 创建工单
按映射表指派给处理人 + 在多维表更新解决方案
24 小时内运维群收到候选知识卡片
运维点击「✅ 通过入库」 → Agent 自动写入 Echo + 标记工单"已沉淀"

8体验官 Agent 落地示意
